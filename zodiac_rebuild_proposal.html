Key Findings

What the analysis revealed.

Examining 17 pages across both generations — including all major victim pages, suspect pages, and a random cross-section — produced four findings that directly shape the rebuild strategy.

Critical

The WordPress victim pages are unfinished stubs. Pages for Paul Stine, Darlene Ferrin, David Faraday, and others literally contain the message "this page is still getting mobile — the original page is here." The WordPress migration was started and stopped. Generation 1 holds the complete, real content for these pages. This is not a problem — it tells us exactly where to focus.

Structure

Hundreds of pages are paginated evidence galleries. Files like BombLetter4.html, BatesCrimeScene4.html, and KaneReport1.html are individual pages of multi-page documents — one image per HTML file, with "next page" links. On desktop with a slow 1990s connection, this made sense. Today, each of these sequences should be a single scrollable page with all images stacked vertically. The content is all there; it just needs to be reassembled.

Opportunity

Structured data is cleanly extractable. The Zodiac calendar, the suspect index pages, and the Domingos/Edwards victim pages all contain structured information in HTML tables. This data can be extracted and rebuilt as proper, mobile-friendly layouts — sortable, linkable, and readable on any screen size.

Discard

The WordPress infrastructure can be discarded entirely. jQuery, the CoBlocks page builder, the GoDaddy "Go" theme stylesheets, Amazon affiliate banners, and old search box widgets account for the bloat but contain no evidence value. None of this needs to migrate.

The Plan

A four-phase extraction and rebuild pipeline.

The rebuild does not require rewriting anything by hand. We build a set of scripts that walk the archive, classify every file, extract the real content, and assemble it into a clean, fast, mobile-ready site. The pipeline has four phases.

1
Classify every HTML file
A script walks all 654 HTML files and assigns each one a type: WordPress article, WordPress stub (unfinished migration), Generation 1 article, Generation 1 image gallery page, Generation 1 navigation page, or WordPress infrastructure (discard). This gives us a complete map of what we're working with before touching anything.
2
Extract content by type
Each type gets its own extraction rule. WordPress articles: pull entry-content, strip the wrapper. Generation 1 articles: strip layout tables, keep text and images. Image gallery sequences: detect the numbering pattern (e.g. BombLetter1 through BombLetter12), group them, and assemble into a single scrollable page per document.
3
Resolve and validate all images
Generation 1 uses relative image paths (images/filename.gif). WordPress uses absolute paths to wp-content/uploads/. Both exist in the archive. The script verifies every image reference resolves to a real file, flags any broken links, and normalizes all paths for the new structure.
4
Build the site structure
All extracted content feeds into a clean, consistent template system: one shared CSS file, section index pages (Victims, Suspects, Evidence, Letters, News), and mobile-first page templates per content type. No WordPress, no plugins, no frameworks — just HTML, CSS, and the evidence itself.
The Result

What the rebuilt site achieves.

The rebuilt zodiackiller.com is not a redesign — it is a restoration and upgrade. The goal is a site that works as well in 2026 as it did in its heyday, on the devices people actually use today.

For visitors

Every page loads in under a second. The cipher images are sharp and legible on a phone screen. The police reports and audio files are accessible without plugins. The suspect and victim pages are navigable with a thumb. The evidence reads the way it was meant to be read.

For Tom

The archive section — the decades of historical material — requires no ongoing maintenance. It is published once and stays. No WordPress updates to apply. No plugin conflicts to resolve. No GoDaddy theme breaking on the next browser release. It simply serves.

The active section — new articles, Today in Zodiac History, ongoing research — gets a simple, clean publishing workflow that Tom controls without needing a developer. The exact format of this tool is a conversation to have once the archive is stable.

Hosting

Where the rebuilt site lives.

The rebuild will produce plain HTML files — the simplest possible thing to host. This opens up several options, each with different tradeoffs. Zodiackiller.com gets real traffic spikes (documentaries, podcasts, news hooks), so resilience under load matters.

GoDaddy (current host)
Replace WordPress with flat HTML files on existing account. Zero migration hassle. Keeps current domain setup. Performance better than WordPress but GoDaddy shared hosting has limits under heavy traffic.
GitHub Pages
Free, version-controlled, easy to update. Every change is tracked and reversible. Soft bandwidth limits (~100 GB/month) — risky during a major news spike. Fine for normal traffic.
Cloudflare Pages Recommended
Free tier. Unlimited bandwidth. Global CDN built in — pages load fast worldwide. Deploys automatically from GitHub. Traffic spikes are absorbed without a second thought. The right infrastructure for a site that gets hit hard without warning.

Our recommendation: GitHub as the repository (version control, edit history, Tom can see every change) with Cloudflare Pages as the delivery layer (free, fast, spike-proof). The two work together seamlessly. Tom pushes an update to GitHub; Cloudflare deploys it automatically within seconds.

The bottom line

Twenty-five years of irreplaceable forensic research should not be held hostage by a WordPress installation and a GoDaddy theme. The evidence is in good shape. The images are intact. The PDFs and audio files are all there.

The fix is not complicated — it is just methodical. We extract the real content from the scaffolding, rebuild it in clean HTML, and publish it somewhere fast and resilient. The archive becomes permanent, stable, and accessible on any device.

The work to get there is scripted and systematic. We are not rebuilding the site by hand — we are building tools that do the rebuilding for us, so the result is consistent across all 600+ pages. Once the pipeline runs, the archive is done. From there, the only ongoing work is Tom's new content, handled by a simple publishing tool built around how he actually works.

The Publishing Tool

How Tom creates and updates pages going forward.

The archive is the foundation — but zodiackiller.com is a living site. Tom publishes new articles, updates suspect pages as evidence develops, runs the Today in Zodiac History series, and adds new research as it emerges. The rebuild needs to support that ongoing work without requiring a developer every time something changes.

The tool we build for this is called the Zodiac Publisher — a dedicated editor that runs on Tom's computer and publishes directly to the live site. No WordPress. No GoDaddy control panel. No FTP client. Just a clean writing environment designed specifically for the way this site works.

How Tom runs it

Primary
Desktop app
Built on the Fluid Fortune Trojan Horse framework — a native desktop wrapper that turns the editor into a proper Mac application. Double-click to open, no browser required, no accidental tab closing. Feels like a real app because it is one. Dependent on resolving a macOS packaging issue currently under review — handled separately.
Fallback
Open in Chrome
The Zodiac Publisher is a single HTML file. If the desktop app isn't ready or isn't behaving, Tom drags the file into Chrome and gets the identical experience — same interface, same features, same Publish button. No degraded mode. No missing features. Same file, two ways to run it.

What the editor looks like

On the left: a list of every page on the site, organized by section — Victims, Suspects, Evidence, Letters, News, Today in Zodiac History. On the right: a simple editor. Tom clicks a page to edit it, or clicks New Page to start fresh. When he's done, he clicks Publish. The page goes live within seconds.

No accounts to manage. No plugins to update. No theme breaking after a WordPress update. The tool is a single file that lives on Tom's Desktop and works entirely offline until the moment he publishes.

What it can do

✍️
Write and edit articles
A clean writing surface with basic formatting — headings, bold, italic, blockquotes for witness statements, and code blocks for cipher text. Nothing more than what the site actually uses. No bloated toolbar.
🖼
Add images and evidence
Drag an image into the editor and it's embedded. The tool handles sizing and mobile formatting automatically. Crime scene photos, suspect images, cipher scans — all handled the same way.
📄
Attach PDFs and audio
Police reports, court documents, and MP3 interview recordings attach with a single click. They appear on the page as readable, accessible links — no plugins required on the visitor's end.
📅
Today in Zodiac History
A dedicated view for the daily history series. Tom sees the calendar, clicks a date, writes the entry, and publishes. Past entries are all there, editable any time. The series lives as a proper navigable archive on the site.
🔗
Update existing pages
Any page on the site — including the migrated archive pages — is editable. If new evidence surfaces about a suspect, Tom opens that suspect's page, makes the change, and publishes. The update is live in seconds.
👁
Preview before publishing
A preview button shows exactly how the page will look on both desktop and mobile before anything goes live. What Tom sees is what visitors get.

What it does not do

The Zodiac Publisher is deliberately narrow in scope. It does not try to be a general-purpose CMS. It has no user accounts, no comment system, no analytics dashboard, no social sharing buttons, no newsletter integration. Every one of those features would add complexity that could break. The tool does exactly what zodiackiller.com needs and nothing else.

How publishing actually works

When Tom clicks Publish, the tool converts his writing into a clean HTML file using the same template as the rest of the site, then pushes it to GitHub — the version-controlled repository where the site lives. Cloudflare Pages detects the new file and deploys it to the live site automatically. The whole process takes about ten seconds.

Every published page is saved in GitHub with a complete history. If Tom publishes something and wants to revert it, the previous version is one click away. Nothing is ever truly lost.

The practical difference for Tom

Right now, publishing a new article on zodiackiller.com means navigating WordPress's block editor, fighting with the GoDaddy theme, hoping images resize correctly on mobile, and waiting for a slow page builder to render. It's friction at every step.

With the Zodiac Publisher: open the tool, write the article, add images, click Publish. The site handles the rest. Mobile formatting is automatic. Images are optimized automatically. The page matches the rest of the site automatically.

The goal is a tool that gets out of Tom's way and lets him focus on the research — which is the part that actually matters.
